# Story 1.1: Project Skeleton and Bootstrap

## Status: Draft

## Story

**As a** developer,
**I want** a working Tauri v2 app with Rust backend, React frontend, logging, and project structure,
**so that** all future features plug into a consistent architecture.

## Acceptance Criteria

1. Project builds cleanly with a Tauri v2 Rust backend (`src-tauri/`) and a React + TypeScript frontend (`src/`).
2. Tauri state management is configured for shared application state across commands.
3. React app uses functional components with hooks and TypeScript strict mode.
4. `tracing` crate writes to both console and rolling file (`logs/dspanel-.log`).
5. Source tree follows the conventions defined in the architecture doc (components, hooks, services, types, etc.).
6. `.editorconfig` enforces TypeScript/Rust coding style and naming conventions.
7. All required cargo crates and npm packages are referenced at their specified versions.
8. Application starts and displays an empty main window with no errors.

## Dev Notes

### Tauri v2 Bootstrap Pattern

The application uses Tauri v2 with a Rust backend for system-level operations and a React frontend for the UI:

```rust
// src-tauri/src/main.rs
use tauri::Manager;

mod commands;
mod state;

fn main() {
    let app_state = state::AppState::new();

    tauri::Builder::default()
        .manage(app_state)
        .invoke_handler(tauri::generate_handler![
            // commands registered here
        ])
        .setup(|app| {
            tracing::info!("DSPanel starting up");
            Ok(())
        })
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

### AppState Skeleton

```rust
// src-tauri/src/state.rs
use std::sync::Mutex;

pub struct AppState {
    pub title: Mutex<String>,
}

impl AppState {
    pub fn new() -> Self {
        Self {
            title: Mutex::new("DSPanel".to_string()),
        }
    }
}
```

The `AppState` struct uses `Mutex` for interior mutability since Tauri state must be `Send + Sync`.

### Source Tree Conventions

Every Rust module in `src-tauri/src/` that contains logic should have corresponding tests (inline `#[cfg(test)]` modules or separate test files). React components and hooks should have corresponding `.test.tsx` or `.test.ts` files:

- `src-tauri/src/services/directory.rs` - tests in `#[cfg(test)] mod tests { ... }`
- `src/components/UserLookup.tsx` - `src/components/UserLookup.test.tsx`
- `src/hooks/useDirectory.ts` - `src/hooks/useDirectory.test.ts`

### Testing

- Rust: `cargo test` with `#[test]` functions, use `cargo-nextest` for parallel execution
- Frontend: vitest with React Testing Library
- Assertions: Rust standard `assert!`, `assert_eq!`; JS/TS via `expect()` from vitest
- Bootstrap tests should verify the app state builds correctly and key services initialize
- Test naming convention: Rust `test_method_name_scenario_expected`, TS `describe/it` blocks
