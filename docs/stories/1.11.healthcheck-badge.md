# Story 1.11: Healthcheck Badge

## Status: Done

## Story

**As a** support technician,
**I want** a visual healthcheck badge on each account,
**so that** I can instantly identify problematic accounts.

## Acceptance Criteria

1. A healthcheck badge is displayed next to the user's name in both the search results list and the detail view.
2. The badge aggregates the following flags: Disabled, Locked, Expired, PasswordExpired, PasswordNeverExpires, Inactive30Days, Inactive90Days, NeverLoggedOn, PasswordNeverChanged.
3. Badge color reflects the worst severity: red for Critical, orange for Warning, green for Healthy.
4. A tooltip on the badge lists all active flags with their descriptions.
5. Badge displays "Healthy" text/icon when no issues are detected.
6. `IHealthCheckService` is a testable service with `EvaluateHealth(DirectoryUser)` method.
7. Comprehensive unit tests cover every flag and all severity combinations.

## Dev Notes

### Architecture

Health check logic is a pure service with no UI dependencies, making it fully unit-testable. The service receives a `DirectoryUser` and returns an `AccountHealthStatus` with all detected flags.

### AccountHealthStatus Model

```csharp
public enum HealthLevel { Healthy, Info, Warning, Critical }

public record HealthFlag(string Name, HealthLevel Severity, string Description);

public class AccountHealthStatus
{
    public HealthLevel Level { get; init; }
    public IReadOnlyList<HealthFlag> ActiveFlags { get; init; } = [];
}
```

### Flag Detection Logic

The service evaluates all flags independently (no short-circuit). If both Inactive30Days and Inactive90Days apply, only Inactive90Days is added (it supersedes the 30-day flag). The overall `HealthLevel` is computed as the maximum severity among all active flags, defaulting to `Healthy` if no flags are active.

### HealthBadge Control

The HealthBadge is a small `UserControl` designed to be embedded inline:

```xml
<controls:HealthBadge HealthStatus="{Binding SelectedUserHealth}" />
```

Internally it uses a `Border` with `CornerRadius` and background bound through `HealthStatusToColorConverter`:

- Healthy: `#4CAF50` (green)
- Info: `#2196F3` (blue)
- Warning: `#FF9800` (orange)
- Critical: `#F44336` (red)

The converter is in `src/DSPanel/Converters/HealthStatusToColorConverter.cs` and maps `HealthLevel` to a `SolidColorBrush`. Theme-aware brush resources can override these defaults.

### Integration with UserLookupViewModel

```csharp
partial void OnSelectedUserChanged(DirectoryUser? value)
{
    SelectedUserHealth = value is not null
        ? _healthCheckService.EvaluateHealth(value)
        : null;
}
```

The `OnSelectedUserChanged` partial method is auto-generated by `[ObservableProperty]` source generator.

### Testing

- Unit tests for `HealthCheckService.EvaluateHealth()`:
  - Each flag in isolation (9 test cases minimum).
  - Multiple flags combined (e.g., Disabled + Locked + PasswordExpired).
  - Severity escalation: Warning flags only -> Warning, any Critical flag -> Critical.
  - Healthy user: enabled, not locked, recent logon, password recently set.
  - Edge cases: `LastLogon` exactly 30 days ago, `AccountExpires` exactly now, null `LastLogon` with `WhenCreated` today.
- Unit tests for `HealthStatusToColorConverter`: each HealthLevel maps to correct brush.
- Test mirror convention: `src/DSPanel/Services/Health/HealthCheckService.cs` -> `src/DSPanel.Tests/Services/Health/HealthCheckServiceTests.cs`.
- Use xUnit 2.9.3 + FluentAssertions 8.8.0.
- Use `TimeProvider` abstraction or injectable clock for deterministic date-based tests.
