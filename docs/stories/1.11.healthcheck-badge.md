# Story 1.11: Healthcheck Badge

## Status: Done

## Story

**As a** support technician,
**I want** a visual healthcheck badge on each account,
**so that** I can instantly identify problematic accounts.

## Acceptance Criteria

1. A healthcheck badge is displayed next to the user's name in both the search results list and the detail view.
2. The badge aggregates the following flags: Disabled, Locked, Expired, PasswordExpired, PasswordNeverExpires, Inactive30Days, Inactive90Days, NeverLoggedOn, PasswordNeverChanged.
3. Badge color reflects the worst severity: red for Critical, orange for Warning, green for Healthy.
4. A tooltip on the badge lists all active flags with their descriptions.
5. Badge displays "Healthy" text/icon when no issues are detected.
6. Health check logic is a testable Rust service with `evaluate_health(user: &DirectoryUser)` method, exposed via Tauri command.
7. Comprehensive unit tests cover every flag and all severity combinations.

## Dev Notes

### Architecture

Health check logic is a pure Rust service with no UI dependencies, making it fully unit-testable. The service receives a `DirectoryUser` and returns an `AccountHealthStatus` with all detected flags. Results are sent to the frontend via a Tauri command.

### Rust Types

```rust
// src-tauri/src/models/health.rs
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Copy, PartialEq, Eq, PartialOrd, Ord, Serialize, Deserialize)]
pub enum HealthLevel {
    Healthy,
    Info,
    Warning,
    Critical,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct HealthFlag {
    pub name: String,
    pub severity: HealthLevel,
    pub description: String,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct AccountHealthStatus {
    pub level: HealthLevel,
    pub active_flags: Vec<HealthFlag>,
}
```

### TypeScript Types

```typescript
// src/types/health.ts
export type HealthLevel = "Healthy" | "Info" | "Warning" | "Critical";

export interface HealthFlag {
  name: string;
  severity: HealthLevel;
  description: string;
}

export interface AccountHealthStatus {
  level: HealthLevel;
  activeFlags: HealthFlag[];
}
```

### Flag Detection Logic

The service evaluates all flags independently (no short-circuit). If both Inactive30Days and Inactive90Days apply, only Inactive90Days is added (it supersedes the 30-day flag). The overall `HealthLevel` is computed as the maximum severity among all active flags, defaulting to `Healthy` if no flags are active.

### HealthBadge Component

```typescript
// src/components/common/HealthBadge.tsx
const levelColors: Record<HealthLevel, string> = {
  Healthy: "#4CAF50",
  Info: "#2196F3",
  Warning: "#FF9800",
  Critical: "#F44336",
};

export function HealthBadge({ healthStatus }: { healthStatus: AccountHealthStatus }) {
  return (
    <span
      className="health-badge"
      style={{ backgroundColor: levelColors[healthStatus.level] }}
      title={healthStatus.activeFlags.map(f => f.description).join("\n")}
    >
      {healthStatus.level === "Healthy"
        ? "Healthy"
        : `${healthStatus.activeFlags.length} issue(s)`}
    </span>
  );
}
```

### Testing

- Rust unit tests for `evaluate_health()`:
  - Each flag in isolation (9 test cases minimum)
  - Multiple flags combined (e.g., Disabled + Locked + PasswordExpired)
  - Severity escalation: Warning flags only -> Warning, any Critical flag -> Critical
  - Healthy user: enabled, not locked, recent logon, password recently set
  - Edge cases: `last_logon` exactly 30 days ago, `account_expires` exactly now, None `last_logon` with `when_created` today
- Frontend tests for HealthBadge: each HealthLevel maps to correct color/text
- Use `cargo test` for Rust tests
- Use vitest + React Testing Library for frontend tests
- Use a clock abstraction or injectable time source for deterministic date-based Rust tests
