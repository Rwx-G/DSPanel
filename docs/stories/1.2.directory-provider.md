# Story 1.2: IDirectoryProvider and AD On-Prem Connection

## Status: Done

## Story

**As a** developer,
**I want** an `IDirectoryProvider` interface with an LDAP implementation,
**so that** all AD queries go through a consistent abstraction.

## Acceptance Criteria

1. `IDirectoryProvider` interface defines methods for searching users, computers, and groups.
2. `LdapDirectoryProvider` implements `IDirectoryProvider` using `System.DirectoryServices.Protocols`.
3. Domain base DN is auto-detected via the `USERDNSDOMAIN` environment variable and confirmed with a `rootDSE` query.
4. Authentication uses the current user's Kerberos credentials (no stored passwords).
5. `LdapDirectoryProvider` is registered in the DI container as a singleton.
6. Unit tests cover all `IDirectoryProvider` methods using mocked LDAP dependencies.
7. Graceful error handling when no domain is available (non-domain-joined machine) - returns empty results and logs a warning.

## Dev Notes

### IDirectoryProvider Interface

```csharp
using DSPanel.Models;

namespace DSPanel.Services.Directory;

public interface IDirectoryProvider
{
    bool IsConnected { get; }
    string? DomainName { get; }
    string? BaseDn { get; }

    Task<bool> TestConnectionAsync();

    Task<IReadOnlyList<DirectoryEntry>> SearchUsersAsync(
        string filter, int maxResults = 50);

    Task<IReadOnlyList<DirectoryEntry>> SearchComputersAsync(
        string filter, int maxResults = 50);

    Task<IReadOnlyList<DirectoryEntry>> SearchGroupsAsync(
        string filter, int maxResults = 50);

    Task<DirectoryEntry?> GetUserByIdentityAsync(string samAccountName);

    Task<IReadOnlyList<DirectoryEntry>> GetGroupMembersAsync(
        string groupDn, int maxResults = 200);
}
```

### DirectoryEntry Model

```csharp
namespace DSPanel.Models;

public sealed class DirectoryEntry
{
    public required string DistinguishedName { get; init; }
    public string? SamAccountName { get; init; }
    public string? DisplayName { get; init; }
    public string? ObjectClass { get; init; }
    public IReadOnlyDictionary<string, string[]> Attributes { get; init; }
        = new Dictionary<string, string[]>();
}
```

This model is intentionally named `DirectoryEntry` within the `DSPanel.Models` namespace. It must not be confused with `System.DirectoryServices.DirectoryEntry` - the project uses `System.DirectoryServices.Protocols` (S.DS.P) which does not have this naming conflict.

### LDAP Connection Pattern

```csharp
using System.DirectoryServices.Protocols;

// Domain detection
var domain = Environment.GetEnvironmentVariable("USERDNSDOMAIN");
if (string.IsNullOrEmpty(domain))
{
    _logger.Warning("USERDNSDOMAIN not set - not domain-joined");
    return;
}

// Connect with Kerberos
var connection = new LdapConnection(
    new LdapDirectoryIdentifier(domain, 389));
connection.AuthType = AuthType.Negotiate;
connection.SessionOptions.ProtocolVersion = 3;
connection.Bind(); // Uses current user's Kerberos ticket

// Discover base DN via rootDSE
var rootDseRequest = new SearchRequest(
    null, // null DN = rootDSE
    "(objectClass=*)",
    SearchScope.Base,
    "defaultNamingContext");
var rootDseResponse = (SearchResponse)connection.SendRequest(rootDseRequest);
var baseDn = rootDseResponse.Entries[0]
    .Attributes["defaultNamingContext"][0].ToString();
```

### LDAP Search Pattern

```csharp
var searchRequest = new SearchRequest(
    baseDn,
    $"(&(objectClass=user)(objectCategory=person)(|(sAMAccountName=*{filter}*)(displayName=*{filter}*)))",
    SearchScope.Subtree,
    "distinguishedName", "sAMAccountName", "displayName", "objectClass",
    "mail", "department", "title", "telephoneNumber");
searchRequest.SizeLimit = maxResults;

var response = (SearchResponse)connection.SendRequest(searchRequest);
```

### Important Notes

- `LdapConnection` is not thread-safe. Use a lock or create connections per request.
- LDAP filters must escape special characters: `*`, `(`, `)`, `\`, NUL. Use a helper method.
- The provider should be registered as singleton but create connections lazily.
- All async methods run LDAP operations on a background thread via `Task.Run` since `System.DirectoryServices.Protocols` is synchronous.

### Testing

- Mock the LDAP layer by extracting connection creation into an injectable factory (`ILdapConnectionFactory`).
- Tests should verify:
  - Correct LDAP filter construction for each search type
  - Proper mapping from `SearchResultEntry` to `Models.DirectoryEntry`
  - Error handling paths (no domain, connection failure, timeout)
- Use `FluentAssertions` for readable assertions
- Test naming convention: `MethodName_Scenario_ExpectedResult`
- Example: `SearchUsersAsync_WhenDomainNotJoined_ReturnsEmptyList`
