# Story 1.3: Permission Level Detection

## Status: Done

## Story

**As a** support technician,
**I want** DSPanel to detect my permission level from AD groups,
**so that** I only see authorized actions.

## Acceptance Criteria

1. `IPermissionService` detects the current user's AD group memberships at startup.
2. Group memberships are mapped to a `PermissionLevel` enum: `ReadOnly`, `HelpDesk`, `AccountOperator`, `DomainAdmin`.
3. AD group names used for detection are configurable (not hardcoded).
4. Permission levels follow inheritance - a `DomainAdmin` implicitly has all lower-level permissions.
5. The current permission level is displayed in the status bar.
6. `HasPermission(PermissionLevel required)` method returns `true` if the user meets or exceeds the required level.
7. Unit tests cover all permission detection and inheritance scenarios.

## Dev Notes

### PermissionLevel Enum

```csharp
namespace DSPanel.Services.Permissions;

public enum PermissionLevel
{
    ReadOnly = 0,
    HelpDesk = 1,
    AccountOperator = 2,
    DomainAdmin = 3
}
```

The numeric values enable simple comparison for inheritance: `currentLevel >= requiredLevel`.

### Default Group Name Mappings

| PermissionLevel | Default AD Group Name |
|----------------|----------------------|
| HelpDesk | DSPanel-HelpDesk |
| AccountOperator | DSPanel-AccountOps |
| DomainAdmin | Domain Admins |

`ReadOnly` is the default level assigned when no matching groups are found. There is no specific AD group for it.

### IPermissionService Interface

```csharp
namespace DSPanel.Services.Permissions;

public interface IPermissionService
{
    PermissionLevel CurrentLevel { get; }
    IReadOnlyList<string> UserGroups { get; }

    Task DetectPermissionsAsync();
    bool HasPermission(PermissionLevel required);
}
```

### Inheritance Logic

The service scans all group memberships and assigns the highest matching level:

```csharp
public bool HasPermission(PermissionLevel required)
{
    return CurrentLevel >= required;
}
```

Detection iterates through all configured group mappings and keeps the maximum:

```csharp
var detectedLevel = PermissionLevel.ReadOnly;

foreach (var group in userGroups)
{
    if (_groupMappings.TryGetValue(group, out var level) && level > detectedLevel)
    {
        detectedLevel = level;
    }
}
```

### Configuration Pattern

Group-to-level mappings should be loaded from configuration so they can be customized per deployment:

```json
{
  "Permissions": {
    "GroupMappings": {
      "DSPanel-HelpDesk": "HelpDesk",
      "DSPanel-AccountOps": "AccountOperator",
      "Domain Admins": "DomainAdmin"
    }
  }
}
```

Use `IOptions<PermissionOptions>` to inject the configuration into `PermissionService`.

### Querying User Groups

The service uses `IDirectoryProvider` to fetch the current user's group memberships. The current user's `sAMAccountName` is obtained from `Environment.UserName`. Groups are read from the `memberOf` attribute of the user object.

### DI Registration

```csharp
services.AddSingleton<IPermissionService, PermissionService>();
```

`DetectPermissionsAsync()` is called during application startup after the `IDirectoryProvider` connection is established.

### Testing

- Mock `IDirectoryProvider` to return controlled group membership lists
- Use `IOptions<PermissionOptions>` with test-specific group mappings
- Test all four permission levels individually
- Test inheritance by verifying `HasPermission` across level boundaries
- Test the fallback to `ReadOnly` when `IDirectoryProvider` is not connected (offline mode)
- Test naming convention: `MethodName_Scenario_ExpectedResult`
