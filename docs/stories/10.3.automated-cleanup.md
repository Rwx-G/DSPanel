# Story 10.3: Automated Cleanup

## Status: Draft

## Story

**As a** DomainAdmin,
**I want** to define cleanup rules that identify and act on stale AD objects,
**so that** I can keep the directory clean with minimal manual effort and full auditability.

## Acceptance Criteria

1. Cleanup rules can target: accounts inactive for X days, accounts never logged on and created more than Y days ago, and accounts disabled for Z days.
2. Available actions per rule: disable account, move to a designated cleanup OU, or delete.
3. Every cleanup operation starts with a mandatory dry-run showing affected objects and proposed actions.
4. An explicit confirmation dialog lists all changes before execution proceeds.
5. All cleanup actions are logged via `IAuditService` with full details.
6. The cleanup feature is gated to DomainAdmin permission level.
7. Unit tests cover rule evaluation, dry-run output, and action execution.

## Tasks / Subtasks

- [ ] Task 1: Create cleanup rule engine (AC: #1)
  - [ ] Create `CleanupRule` model (name, condition type, threshold days, action type, target OU)
  - [ ] Create `CleanupRuleEvaluator` that queries AD and returns matching objects
  - [ ] Condition types: InactiveDays, NeverLoggedOnCreatedDays, DisabledDays
- [ ] Task 2: Rule configuration UI (AC: #1, #2)
  - [ ] `CleanupRulesView.xaml` with rule list and create/edit/delete
  - [ ] Rule editor: condition type picker, threshold input, action picker, target OU picker
  - [ ] Persist rules in settings (JSON)
- [ ] Task 3: Dry-run execution (AC: #3)
  - [ ] Execute rule evaluation without making changes
  - [ ] Display results in a DataGrid: object DN, current state, proposed action
  - [ ] Show summary counts (total affected, by action type)
- [ ] Task 4: Confirmation workflow (AC: #4)
  - [ ] Confirmation dialog listing all proposed changes
  - [ ] Require explicit "Execute" button click (not just OK)
  - [ ] Option to deselect individual objects before execution
- [ ] Task 5: Action execution (AC: #2, #5)
  - [ ] Disable: set userAccountControl ACCOUNTDISABLE flag
  - [ ] Move: LDAP ModifyDN to cleanup OU
  - [ ] Delete: LDAP delete operation
  - [ ] Log each action via `IAuditService`
  - [ ] Report execution results (success/failure per object)
- [ ] Task 6: Permission gating (AC: #6)
  - [ ] Gate cleanup navigation to DomainAdmin level
- [ ] Task 7: Write tests (AC: #7)
  - [ ] Test rule evaluation returns correct matching objects
  - [ ] Test dry-run does not modify AD
  - [ ] Test each action type executes correct LDAP operation
  - [ ] Test audit logging is called for each action
  - [ ] Test permission gating

## Dev Notes

### Rule Persistence

Rules are stored as JSON in the application settings:

```json
{
  "cleanupRules": [
    {
      "name": "Disable inactive 180 days",
      "conditionType": "InactiveDays",
      "thresholdDays": 180,
      "actionType": "Disable",
      "targetOU": null
    }
  ]
}
```

### Safety Considerations

- Dry-run is mandatory and cannot be skipped.
- Delete actions require double confirmation.
- Built-in accounts (Administrator, Guest, krbtgt) are always excluded from cleanup.
- Service accounts (identified by naming convention or OU) can be excluded via configuration.
