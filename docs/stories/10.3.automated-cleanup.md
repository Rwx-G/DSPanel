# Story 10.3: Automated Cleanup

## Status: Done

## Story

**As a** DomainAdmin,
**I want** to define cleanup rules that identify and act on stale AD objects,
**so that** I can keep the directory clean with minimal manual effort and full auditability.

## Acceptance Criteria

1. Cleanup rules can target: accounts inactive for X days, accounts never logged on and created more than Y days ago, and accounts disabled for Z days.
2. Available actions per rule: disable account, move to a designated cleanup OU, or delete.
3. Every cleanup operation starts with a mandatory dry-run showing affected objects and proposed actions.
4. An explicit confirmation dialog lists all changes before execution proceeds.
5. All cleanup actions are logged via `AuditService` trait with full details.
6. The cleanup feature is gated to DomainAdmin permission level.
7. Unit tests cover rule evaluation, dry-run output, and action execution.

## Dev Notes

### Rule Persistence

Rules are stored as JSON in the application settings:

```json
{
    "cleanupRules": [
        {
            "name": "Disable inactive 180 days",
            "conditionType": "InactiveDays",
            "thresholdDays": 180,
            "actionType": "Disable",
            "targetOU": null
        }
    ]
}
```

### Safety Considerations

- Dry-run is mandatory and cannot be skipped.
- Delete actions require double confirmation.
- Built-in accounts (Administrator, Guest, krbtgt) are always excluded from cleanup.
- Service accounts (identified by naming convention or OU) can be excluded via configuration.
