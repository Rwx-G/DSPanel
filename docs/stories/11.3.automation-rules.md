# Story 11.3: Trigger-Based Automation

## Status: Draft

## Story

**As a** DomainAdmin,
**I want** to define automation rules with triggers and actions,
**so that** routine AD tasks are executed automatically when specific conditions are met.

## Acceptance Criteria

1. A rule editor allows defining rules with a trigger condition and one or more actions.
2. Supported triggers: object created, object moved, group membership changed, specific attribute changed.
3. Supported actions: apply preset, add to group, remove from group, send notification, run external script.
4. Rules can be individually enabled or disabled.
5. All automation executions are logged via `AuditService` trait.
6. A dry-run mode simulates rule execution without making changes.
7. The automation feature is gated to DomainAdmin permission level.
8. Unit tests cover trigger detection, action execution, and dry-run behavior.

## Dev Notes

### Polling Strategy

Trigger detection uses a polling approach (configurable interval, default 5 minutes) rather than real-time event subscription. Each poll:

1. Query AD for objects modified since last poll (using `whenChanged` or USN tracking).
2. Compare attribute values against stored state.
3. Evaluate rule triggers against detected changes.
4. Execute matched actions.

### Rule Persistence

```json
{
  "automationRules": [
    {
      "id": "rule-001",
      "name": "New user setup",
      "enabled": true,
      "trigger": {
        "type": "ObjectCreated",
        "scope": "OU=Users,DC=contoso,DC=com",
        "objectClass": "user"
      },
      "actions": [
        { "type": "ApplyPreset", "presetName": "New Employee Onboarding" },
        { "type": "AddToGroup", "groupDn": "CN=All-Users,OU=Groups,DC=contoso,DC=com" }
      ]
    }
  ]
}
```
