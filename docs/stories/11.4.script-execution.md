# Story 11.4: External Script Execution

## Status: Draft

## Story

**As a** DomainAdmin,
**I want** to run external scripts (PowerShell, executables) against AD objects from within DSPanel,
**so that** I can extend DSPanel's capabilities with custom automation without modifying the application.

## Acceptance Criteria

1. A "Run Script" option is available in the context menu of user, group, and computer objects.
2. A script library is configured in settings by specifying a folder path containing scripts.
3. AD object attributes (SAMAccountName, DN, email, etc.) are passed to the script as parameters.
4. Script output (stdout and stderr) is captured and displayed in an output panel.
5. Scripts run with the current user's credentials (no elevation).
6. All script executions are logged via `IAuditService`.
7. The feature is gated to DomainAdmin permission level.
8. Unit tests cover the script execution service, parameter mapping, and output capture.

## Tasks / Subtasks

- [ ] Task 1: Create `IScriptExecutionService` interface (AC: #1, #3, #4)
  - [ ] Define `ExecuteAsync(string scriptPath, Dictionary<string, string> parameters)` returning `ScriptResult`
  - [ ] Create `ScriptResult` model (exitCode, stdout, stderr, duration)
- [ ] Task 2: Implement script execution (AC: #3, #4, #5)
  - [ ] Use `System.Diagnostics.Process` to launch scripts
  - [ ] Configure `ProcessStartInfo`: UseShellExecute=false, RedirectStandardOutput=true, RedirectStandardError=true, CreateNoWindow=true
  - [ ] Pass parameters as command-line arguments
  - [ ] Support `.ps1` (via powershell.exe), `.bat`, `.cmd`, and `.exe` files
  - [ ] Implement execution timeout (configurable, default 60 seconds)
- [ ] Task 3: Create script picker dialog (AC: #1, #2)
  - [ ] Scan configured script folder for supported file types
  - [ ] Display script list with name and file type
  - [ ] Allow selecting a script to run against the current object
- [ ] Task 4: Parameter mapping (AC: #3)
  - [ ] Map common AD attributes to script parameters:
    - `-SAMAccountName`, `-DistinguishedName`, `-Email`, `-DisplayName`, `-Department`
  - [ ] Also set as environment variables: `DSPANEL_SAM`, `DSPANEL_DN`, `DSPANEL_EMAIL`, etc.
- [ ] Task 5: Output panel (AC: #4)
  - [ ] Modal or docked panel showing real-time script output
  - [ ] Separate stdout (white) and stderr (red) display
  - [ ] Show exit code and execution duration on completion
  - [ ] Copy output to clipboard button
- [ ] Task 6: Audit logging (AC: #6)
  - [ ] Log script path, target object, parameters, exit code, and duration via `IAuditService`
  - [ ] Do not log full output (may contain sensitive data)
- [ ] Task 7: Permission gating (AC: #7)
  - [ ] Gate "Run Script" context menu to DomainAdmin level
- [ ] Task 8: Write tests (AC: #8)
  - [ ] Test parameter mapping produces correct arguments
  - [ ] Test process launch configuration
  - [ ] Test output capture (stdout and stderr)
  - [ ] Test timeout handling
  - [ ] Test audit logging on execution

## Dev Notes

### Process Configuration

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = scriptPath.EndsWith(".ps1")
        ? "powershell.exe"
        : scriptPath,
    Arguments = scriptPath.EndsWith(".ps1")
        ? $"-ExecutionPolicy Bypass -File \"{scriptPath}\" {argsString}"
        : argsString,
    UseShellExecute = false,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    CreateNoWindow = true
};
```

### Security Considerations

- Scripts run under the current user's Windows identity (no impersonation or elevation).
- The script folder path must be validated to exist and be accessible.
- Only supported file extensions (.ps1, .bat, .cmd, .exe) are shown in the picker.
- Script execution is always explicit (user-initiated, never automatic).
