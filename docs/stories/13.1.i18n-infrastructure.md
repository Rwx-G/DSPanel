
### i18n Initialization

```typescript
// src/i18n.ts
import i18n from "i18next";
import { initReactI18next } from "react-i18next";
import en from "./locales/en.json";

i18n.use(initReactI18next).init({
  resources: { en: { translation: en } },
  lng: "en",
  fallbackLng: "en",
  interpolation: { escapeValue: false },
});

export { i18n };
```

### Key Naming Convention

- `{view}.{element}` pattern: `userLookup.title`, `passwordReset.successMessage`
- `common.*` for shared strings: `common.save`, `common.cancel`
- `sidebar.*` for navigation labels
- `errors.*` for error messages

### Security Review

No security concerns. The i18n system does not handle user input in translation keys (no injection risk). `interpolation.escapeValue` is `false` which is correct for React (React handles escaping). Language persistence uses the existing AppSettings IPC channel.

### Performance Considerations

No concerns. All translations are bundled statically (no lazy loading needed at current scale). `Intl` formatters are created per-call which is fine for the usage patterns. If performance becomes a concern, formatter instances could be cached by locale.
