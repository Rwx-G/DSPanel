# Story 14.1: Security Indicators Backend Foundation

## Status: Ready for Review

## Story

**As a** DSPanel developer,
**I want** a pure Rust service that evaluates per-object security indicators from already-fetched LDAP attributes,
**so that** the UI layer can render the indicators without performing additional LDAP round-trips and the logic can be unit-tested deterministically.

## Acceptance Criteria

1. New service module `src-tauri/src/services/security_indicators.rs` exposes a pure function `evaluate_user_indicators(user: &DirectoryUser) -> SecurityIndicatorSet` and an analogous `evaluate_computer_indicators(computer: &DirectoryComputer) -> SecurityIndicatorSet`.
2. `SecurityIndicatorSet` carries an ordered list of `SecurityIndicator { kind: SecurityIndicatorKind, severity: IndicatorSeverity, description_key: String, metadata: Option<serde_json::Value> }` where `kind` is a stable enum, `description_key` is an i18n key (not a translated string), and `metadata` is an optional structured payload that lets specific indicators carry additional context for the UI to render in tooltips (e.g. for `Rbcd` and `ConstrainedDelegation` the parsed list of allowed principal SIDs / target SPNs - see Story 14.3).
3. `IndicatorSeverity` reuses the existing `HealthLevel` enum (`Healthy`, `Info`, `Warning`, `Critical`) for visual consistency with the existing `HealthBadge`. No new severity scale is introduced.
4. The service detects all eight indicators specified in stories 14.2 and 14.3 from `userAccountControl` bit flags (`PasswordNotRequired` 0x0020, `ReversibleEncryption` 0x0080, `PasswordNeverExpires` 0x10000, `DontRequirePreauth` 0x400000, `UnconstrainedDelegation` 0x80000) and from attribute presence (`servicePrincipalName`, `msDS-AllowedToDelegateTo`, `msDS-AllowedToActOnBehalfOfOtherIdentity`).
5. A new Tauri command `evaluate_security_indicators` exposes the service to the frontend, accepting either a `DirectoryUser` or a `DirectoryComputer` payload.
6. All required attributes (`userAccountControl`, `servicePrincipalName`, `msDS-AllowedToDelegateTo`, `msDS-AllowedToActOnBehalfOfOtherIdentity`) are already in the `USER_ATTRS` / `COMPUTER_ATTRS` arrays in `services/ldap_directory.rs` - no new LDAP requests are added by this epic.
7. Unit tests cover every indicator in isolation, multiple indicators combined on a single object, and the empty (no indicators) case. Tests follow the same pattern as `services/health.rs::tests`.
8. Severity escalation rule: `Kerberoastable` and `PasswordNeverExpires` are upgraded from `Warning` to `Critical` when the user is also `adminCount=1` (AdminSDHolder-protected). The escalation logic is part of this story's foundation service so that all consumers see consistent severity, and it is covered by dedicated unit tests for every combination.

## Dev Notes

### Architecture

This service is a **pure function** layer with **zero dependencies on the LDAP connection**. It only reads attributes that are already fetched and serialized into `DirectoryUser` / `DirectoryComputer`. This keeps the implementation deterministic, fast, and trivially testable. The same pattern is used by `services/health.rs::evaluate_health` (Story 1.11), which the QA team validated as the canonical pattern for object-level evaluators.

### Why a foundation story

Stories 14.2 (user badges) and 14.3 (computer badges) need the `SecurityIndicatorSet` type and the evaluator functions. Stories 14.4-14.6 (quick-fixes) need the same enums to know which badge to refresh after a fix. Splitting this foundation out lets the three downstream story tracks proceed in parallel and keeps each downstream story focused on UI / IPC concerns rather than mixing in detection logic.

### Rust types

```rust
// src-tauri/src/services/security_indicators.rs
use serde::{Deserialize, Serialize};
use crate::models::{DirectoryUser, DirectoryComputer};
use crate::services::health::HealthLevel;

pub type IndicatorSeverity = HealthLevel;

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum SecurityIndicatorKind {
    Kerberoastable,
    PasswordNotRequired,
    PasswordNeverExpires,
    ReversibleEncryption,
    AsRepRoastable,
    UnconstrainedDelegation,
    ConstrainedDelegation,
    Rbcd,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SecurityIndicator {
    pub kind: SecurityIndicatorKind,
    pub severity: IndicatorSeverity,
    /// i18n key. UI translates via `t(description_key)`. Never a translated string.
    pub description_key: String,
    /// Optional structured payload for indicators that carry extra context.
    /// `Rbcd` populates `{"allowed_principals": ["S-1-...", ...]}` from the
    /// parsed `msDS-AllowedToActOnBehalfOfOtherIdentity` security descriptor.
    /// `ConstrainedDelegation` populates `{"target_spns": ["http/web1", ...]}`
    /// from `msDS-AllowedToDelegateTo`. Other indicators set this to `None`.
    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub metadata: Option<serde_json::Value>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SecurityIndicatorSet {
    pub indicators: Vec<SecurityIndicator>,
    pub highest_severity: IndicatorSeverity,
}
```

### UAC bit flag reference

```
PasswordNotRequired       = 0x00000020
ReversibleEncryption      = 0x00000080
PasswordNeverExpires      = 0x00010000
UnconstrainedDelegation   = 0x00080000  (TRUSTED_FOR_DELEGATION)
DontRequirePreauth        = 0x00400000  (AS-REP roastable)
```

These constants must be declared at the top of `security_indicators.rs` as `const` for clarity. Do not duplicate them across files - if `ldap_directory.rs` already has any of these, import them from there or move them to a shared `models/uac.rs` module.

### Severity escalation contract

Two indicators escalate from `Warning` to `Critical` when the target user is AdminSDHolder-protected (`adminCount = 1`):
- `Kerberoastable`: a kerberoastable Domain Admin is the highest-impact kerberoastable target, since cracking the SPN ticket recovers a Tier 0 password.
- `PasswordNeverExpires`: a never-expiring Domain Admin password is the textbook example of the AdminSDHolder hardening anti-pattern.

The other indicators do not escalate because they are already at their maximum severity (Critical) regardless of admin status.

### i18n description keys

Each indicator carries a `description_key` such as `securityIndicators.kerberoastable.description`, `securityIndicators.passwordNotRequired.description`, etc. The actual translations live in story 14.2 / 14.3 deliverables (per-namespace JSON files); this story only emits the keys.

### Completion Notes List

- COMPUTER_ATTRS was missing both `msDS-AllowedToDelegateTo` and `msDS-AllowedToActOnBehalfOfOtherIdentity` - added both. Existing callers see two extra entries in `entry.attributes`, no behavioral change.
- `search_entry_to_directory_entry` previously stripped binary attributes (only `se.attrs` was copied, `se.bin_attrs` was discarded). To support RBCD detection without an extra LDAP read, added a `BINARY_ATTRS_TO_SURFACE` whitelist that base64-encodes the listed binary attributes into the standard `attributes` map under their original key. Currently lists only `msDS-AllowedToActOnBehalfOfOtherIdentity`; future binary-attr indicators can extend the list.
- `sid_bytes_to_string` in `services/ldap_directory.rs` was promoted from private to `pub` so the new `extract_dacl_principals` helper in `services/dacl.rs` can format SIDs without duplicating the binary-to-string conversion.
- New `extract_dacl_principals(sd_bytes) -> Result<Vec<String>>` helper in `services/dacl.rs` walks the DACL of a binary security descriptor and returns deduplicated SID strings of every Allow-type ACE trustee. Handles ACCESS_ALLOWED_ACE (0x00) and ACCESS_ALLOWED_OBJECT_ACE (0x05); skips deny / audit ACEs.
- Indicator service signature deviates slightly from the story Dev Notes: the evaluators take typed projection structs (`UserIndicatorInput`, `ComputerIndicatorInput`) rather than `&DirectoryUser` / `&DirectoryComputer` directly. There is no `DirectoryUser` Rust type (only the generic `DirectoryEntry` + the typed `DirectoryUser` on the frontend). The projection-input pattern aligns with the existing `evaluate_health` / `HealthInput` design, which the QA team validated as canonical (Story 1.11 review).
- Four Tauri commands exposed (single + batch for user and computer) instead of two, mirroring `evaluate_health_cmd` / `evaluate_health_batch` pattern. The batch variants let the user/computer lookup pages compute badges for an entire visible page in one IPC call.
- The RBCD test constructs a minimal SECURITY_DESCRIPTOR_RELATIVE blob byte-by-byte to exercise the parser end-to-end (base64 decode + SD walk + SID extraction).
