# Story 14.2: User Security Indicator Badges

## Status: Ready for Review

## Story

**As an** AD operator,
**I want** visual badges on users that flag known dangerous configurations (kerberoastable, password not required, password never expires, reversible encryption, AS-REP roastable),
**so that** I see the risk at the moment I am looking at the account, without having to consult a separate audit tool.

## Acceptance Criteria

1. The `UserDetail` page renders a row of badges next to the existing `HealthBadge`, `enabled/disabled` and `locked` badges.
2. Five user-side indicators render their own badge: `Kerberoastable` (severity Warning - Critical if also adminCount=1), `PasswordNotRequired` (Critical), `PasswordNeverExpires` (Warning - Critical if also adminCount=1), `ReversibleEncryption` (Critical), `AsRepRoastable` (Critical).
3. Each badge uses the existing `StatusBadge` component with the variant matching its severity (no new component is introduced).
4. Each badge has a tooltip on hover that explains the attack vector in non-technical operator language and references the AD attribute that triggered it (e.g. "userAccountControl bit DONT_REQUIRE_PREAUTH").
5. Translations exist in `src/locales/{en,fr,de,it,es}/userDetail.json` (or the existing namespace used by UserDetail) for the badge label and the tooltip body of every indicator.
6. The user search results list (`UserLookup`) renders only the **highest severity** indicator as a small dot next to the existing health dot, to keep the row compact. Hovering the dot opens a popover listing all indicators on the user.
7. Frontend unit tests verify badge rendering for each indicator and the dot aggregation in the lookup list.

## Dev Notes

### Architecture

This story is purely a UI consumer of the Story 14.1 backend. It does not duplicate detection logic. If a future regulation or a customer report requires a new indicator, only Story 14.1 needs to be touched - this story's UI just iterates whatever indicators come back. That separation is the point of the foundation story.

### Reusing existing patterns

- **Badge rendering**: `StatusBadge` from Story 1.6 (Common Controls). Two existing variants we need: `warning` (yellow/orange) for `Warning` severity, `error` (red) for `Critical`. The component already exists, do not create new badge primitives.
- **Tooltip**: native `title` attribute is acceptable for short tooltips. For richer tooltip content (multi-line, formatted), reuse the popover pattern from `GroupHygiene` info buttons (referenced in `.claude/CLAUDE.md` UI rules section).
- **Batch evaluation in lookup**: same pattern as `healthMap` state in `UserLookup.tsx` (Story 1.11 QA Results): `Promise.all(users.map(u => invoke('evaluate_user_security_indicators', { user: u })))` and store in a `Map`.

### Severity-to-variant mapping

```typescript
const INDICATOR_TO_BADGE_VARIANT: Record<SecurityIndicatorKind, "warning" | "error"> = {
  Kerberoastable: "warning",       // becomes "error" if escalated; backend tells us
  PasswordNotRequired: "error",
  PasswordNeverExpires: "warning", // becomes "error" if escalated
  ReversibleEncryption: "error",
  AsRepRoastable: "error",
};
```

Important: do NOT compute the variant from the indicator `kind` alone. Always read it from `indicator.severity`:

```typescript
const variant: "warning" | "error" = indicator.severity === "Critical" ? "error" : "warning";
```

This way the AdminSDHolder escalation in Story 14.1 surfaces correctly without the UI re-implementing the rule.

- 1-2 sentences max

### Lookup list dot design

```typescript
// src/components/common/SecurityIndicatorDot.tsx
interface Props {
  indicators: SecurityIndicatorSet;
}
export function SecurityIndicatorDot({ indicators }: Props) {
  if (indicators.indicators.length === 0) return null;
  const color = indicators.highest_severity === "Critical"
    ? "var(--color-error)"
    : "var(--color-warning)";
  // dot + popover-on-hover listing each indicator
}
```

The dot must be visually distinct from the existing health dot so a user can tell them apart at a glance. Suggestion: small shield icon instead of a plain dot.

### Completion Notes List

- Backend touch-up `fix(security): rename_all camelCase on SecurityIndicator wire payload` (commit f017f16). Story 14.1 left the SecurityIndicator struct without `#[serde(rename_all = "camelCase")]` while the enclosing SecurityIndicatorSet had it, which would have forced the TS types into a mixed snake / camel payload (`description_key` next to `highestSeverity`). Added the attribute so the wire format is uniformly camelCase. 21 backend tests still pass.
- TS types use `descriptionKey` (camelCase) consistent with the rest of the project after the backend fix. The metadata field is `metadata?: Record<string, unknown>` because RBCD / ConstrainedDelegation populate it via `serde_json::json!()` literals (snake_case keys inside) - documented inline.
- `severityToBadgeVariant(severity)` exported instead of a kind-keyed `INDICATOR_TO_BADGE_VARIANT` map. Reading the per-indicator severity (which the backend already escalates for Kerberoastable / PasswordNeverExpires when `adminCount=1`) is the only correct way to get the variant - a kind-keyed map would have re-implemented the rule incorrectly in the UI.
- UserDetail receives `securityIndicators` as a prop from UserLookup rather than calling `invoke()` itself. This matches the existing healthStatus data flow and avoids per-detail-render IPC churn.
- Created a new locale namespace `securityIndicators.json` (5 languages) instead of extending `userDetail.json` because the keys (badge + tooltip per indicator + dot helpers) form a self-contained vocabulary that will also be consumed by Story 14.3 (computer badges) and the future Stories 14.4-14.6 (quick-fix dialogs).
- SecurityIndicatorDot uses Shield (Warning) / ShieldAlert (Critical) icons instead of a plain colored dot, so it is visually distinct from the existing HealthBadge dot at a glance per Story 14.2 Dev Notes "Lookup list dot design" suggestion.
- `refreshSelectedUser` now wraps health + indicator evaluation in `Promise.all` so quick-fix actions (Stories 14.4-14.6) refresh both maps in a single round-trip.
