# Story 14.3: Computer Security Indicator Badges

## Status: Ready for Review

## Story

**As an** AD operator,
**I want** visual badges on computer objects that flag dangerous Kerberos delegation configurations,
**so that** I can identify computers that are prime targets for Kerberos relay or constrained-delegation abuse before scheduling maintenance or decommissioning them.

## Acceptance Criteria

1. The `ComputerDetail` page (and equivalent area in `ComputerLookup`) renders a row of badges following the same pattern as Story 14.2.
2. Three computer-side indicators render their own badge: `UnconstrainedDelegation` (Critical - any compromise of this computer compromises every account that ever authenticated against it), `ConstrainedDelegation` (Warning, with the list of target SPNs in the tooltip), `RBCD` (Warning, with the list of allowed-to-act SIDs in the tooltip).
3. The tooltips list the actual values of `msDS-AllowedToDelegateTo` and `msDS-AllowedToActOnBehalfOfOtherIdentity` for the operator to verify what the configuration actually permits.
4. Translations are added to the relevant computer namespace in `src/locales/{en,fr,de,it,es}/`.
5. The computer lookup list mirrors the user-list dot pattern: highest severity dot, popover on hover listing all indicators.
6. Frontend unit tests cover the three indicator badges and the dot aggregation.

## Dev Notes

### Architecture

Mirrors Story 14.2 on the computer side. The same `SecurityIndicator` / `SecurityIndicatorSet` types, the same `SecurityIndicatorDot` component, the same i18n key layout. Only the kinds differ.

### RBCD parsing - resolved during PO review

The attribute `msDS-AllowedToActOnBehalfOfOtherIdentity` is a binary security descriptor blob, not a plain SID list. The PO review on 2026-04-26 resolved the ambiguity: Story 14.1 was updated to expose the parsed principal SIDs via `SecurityIndicator.metadata = Some({ "allowed_principals": [...] })`, populated by the existing Rust SD parser in `services/dacl.rs`. This story therefore reads the field directly with no parsing logic on the frontend, and no TypeScript SD parser is needed.

The same pattern applies to `ConstrainedDelegation` whose `msDS-AllowedToDelegateTo` is a multi-valued string attribute (no parsing needed). The metadata field standardizes the contract with the UI: the renderer always reads `indicator.metadata`, irrespective of whether the underlying AD attribute is structured or plain.

- `StatusBadge`, `SecurityIndicatorDot`, popover-on-hover - all from Story 14.2

### Severity rationale

- `UnconstrainedDelegation` = Critical: compromise of this single host compromises any account that ever authenticates against it. Equivalent to "Tier 0 by reachability".
- `ConstrainedDelegation` = Warning: more limited blast radius (only the listed target SPNs). Operator should verify the list is intentional and minimal.
- `RBCD` = Warning: the listed principals can impersonate users to this computer. Often legitimate (e.g. SQL Server delegation) but can be abused if a low-privilege account has write to `msDS-AllowedToActOnBehalfOfOtherIdentity`.

### Code Quality Assessment

Story 14.3 is a near-perfect mirror of Story 14.2 on the computer side - same evaluator service pattern, same prop-based data flow, same SecurityIndicatorDot reuse, same i18n-in-5-languages-per-commit discipline. The implementation closes the foundational gap that DirectoryComputer was missing `rawAttributes` (DirectoryUser had it; the type asymmetry would have made the indicator service impossible) and adds it cleanly with mapEntryToComputer pass-through. The metadata-aware tooltip is the one new wrinkle: a small `tooltipParamsFor` helper extracts `target_spns` for ConstrainedDelegation and `allowed_principals` for Rbcd from the indicator metadata, narrows via `Array.isArray`, and joins for display. The helper returns `{}` for kinds without metadata - safe to always call because i18next ignores extra interpolation params.

The dev agent rejected two Dev Notes proposals in favor of better choices, both documented:
1. Differentiated i18n keys (`tooltip_with_targets` / `tooltip_with_principals`) → kept a single `tooltip` key per kind, consistent with the user-side namespace and uniform render code via `tooltipParamsFor`.
2. RBCD i18n key as `RBCD` (per the original story Dev Notes) → corrected to `Rbcd` matching the SecurityIndicatorKind PascalCase enum value (Rust serde default). The all-caps version would have caused a silent translation miss.

The QA-14.2 concern about escaping the metadata principal/SPN list strings is fully addressed by architecture, not by escaping logic: the joined string flows into `title={...}` (React's standard JSX prop), which goes through setAttribute. setAttribute always treats the value as text - no HTML parsing path. No `innerHTML` or `dangerouslySetInnerHTML` exists in either ComputerDetail.tsx or ComputerLookup.tsx (verified by grep). Even an attacker-controlled SID like `S-1-5-21-' onmouseover='alert(1)` would render as a literal text title with no script execution.
