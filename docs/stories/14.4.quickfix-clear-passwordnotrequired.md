# Story 14.4: Quick-Fix - Clear PasswordNotRequired

## Status: Draft

## Story

**As an** AccountOperator,
**I want** a "Clear PasswordNotRequired flag" button next to the `PasswordNotRequired` badge,
**so that** I can remediate the misconfiguration in two clicks (open user, click button) instead of switching to ADUC, finding the user, opening properties, navigating to Account tab, unticking the box, and applying.

## Acceptance Criteria

1. The badge `PasswordNotRequired` on `UserDetail` (Story 14.2) gains an inline button "Fix" visible only when the operator's permission level is `AccountOperator` or higher.
2. Clicking the button opens a `ConfirmationDialog` (existing component) with the i18n message explaining: what the flag does, why it is dangerous, what the fix does (`userAccountControl &= ~0x0020`), and a checkbox "I understand this user must comply with the domain password policy at next logon".
3. The dialog is gated by the existing `MfaGate` from Epic 2 if the operator has MFA configured and the action category is set to require MFA in the MFA settings.
4. On confirmation, a new Tauri command `clear_password_not_required` is invoked on the user DN.
5. The backend command reads the current `userAccountControl`, clears bit 0x0020, writes it back via `Mod::Replace`, and emits an `AuditEntry` with action `clear_password_not_required`, target_dn = the user DN, and details listing the previous and new UAC value.
6. The audit entry is captured by the existing SHA-256 hash chain (Epic 11) and forwarded to syslog if configured.
7. After successful fix, the `UserDetail` view reloads the user attributes and the `PasswordNotRequired` badge disappears.
8. Backend command is permission-gated at the service layer: invocation by a `ReadOnly` or `HelpDesk` operator returns an `AppError::PermissionDenied` even if the IPC call is somehow forged.
9. Unit tests cover the UAC bit manipulation (with multiple combinations of other UAC flags set) and the permission gate.

## Dev Notes

### Architecture

This is a focused write operation on a single LDAP attribute. The complexity is not in the LDAP write itself (single-bit flip on `userAccountControl`) but in the **safety wrapping**:

- Permission gate at the service layer (defense in depth - frontend hides the button, backend rejects the call)
- MfaGate at the UI layer for high-trust environments
- Object snapshot before the write (rollback capability)
- Audit trail with hash chain (tamper evidence)
- Idempotent no-op when the flag is already clear (avoid spurious audit entries)

### Why AccountOperator and not HelpDesk

The `PasswordNotRequired` flag affects whether the user must comply with the password policy. HelpDesk operators in DSPanel are scoped to user lookups, password resets, and unlocks - changing UAC bits is outside their remit per the existing permission matrix in `permissions.rs`. AccountOperator is the right level (matches the AD built-in group of the same name).

### Idempotence note

If two operators independently click "Fix" within seconds, the second invocation reads the new UAC, finds the bit already cleared, and returns Ok without writing or auditing. This is intentional to avoid both spurious audit entries and a useless second LDAP write. The audit log only records actual state changes.

### Rust signature

```rust
// src-tauri/src/commands/account.rs

#[tauri::command]
pub async fn clear_password_not_required(
    state: State<'_, AppState>,
    user_dn: String,
) -> Result<(), AppError> {
    clear_password_not_required_inner(&state, &user_dn).await
}

pub(crate) async fn clear_password_not_required_inner(
    state: &AppState,
    user_dn: &str,
) -> Result<(), AppError> {
    if !state
        .permission_service
        .has_permission(PermissionLevel::AccountOperator)
    {
        return Err(AppError::PermissionDenied(
            "Clearing PasswordNotRequired requires AccountOperator permission".to_string(),
        ));
    }

    let provider = state.provider();
    let current_uac = provider
        .get_user_account_control(user_dn)
        .await
        .map_err(|e| AppError::Directory(e.to_string()))?;

    let new_uac = current_uac & !0x0020u32;
    if new_uac == current_uac {
        return Ok(()); // idempotent no-op
    }

    capture_snapshot(state, user_dn, "clear_password_not_required");

    provider
        .modify_attribute(user_dn, "userAccountControl", &[new_uac.to_string()])
        .await
        .map_err(|e| AppError::Directory(e.to_string()))?;

    state.audit_service.record(AuditEntry {
        operator: state.audit_service.operator(),
        action: "clear_password_not_required".to_string(),
        target_dn: user_dn.to_string(),
        details: format!("uac: 0x{:X} -> 0x{:X}", current_uac, new_uac),
        success: true,
        timestamp: now_iso(),
    });

    Ok(())
}
```

### MfaGate integration

The `<MfaGate action="...">` component (Story 2.5) consults the operator's MFA settings and prompts for a TOTP code when the action requires MFA. Pass `action="clear_password_not_required"` so the MFA configuration UI (Story 2.5 / Story 12.x) can selectively require MFA for this fix.
