# Story 14.5: Quick-Fix - Remove Unused SPN

## Status: Draft

## Story

**As an** AccountOperator,
**I want** to be able to remove a Service Principal Name from a user account in one click after reviewing the list of registered SPNs,
**so that** I can mitigate kerberoasting risk on a service account whose SPN is leftover from a decommissioned service.

## Acceptance Criteria

1. The `Kerberoastable` badge on `UserDetail` (Story 14.2) gains an inline button "Manage SPNs" visible only when the operator's permission level is `AccountOperator` or higher.
2. Clicking opens a dialog listing every value of `servicePrincipalName` on the user with a checkbox next to each. The dialog explains the kerberoasting attack and recommends removing SPNs that no longer correspond to a running service.
3. The operator can select one or more SPNs to remove and click "Remove selected". The dialog goes through `MfaGate` if applicable.
4. A new Tauri command `remove_user_spns` accepts the user DN and a list of SPN strings to remove.
5. The backend command reads the current `servicePrincipalName` multi-valued attribute, removes the listed values, and writes the remaining set back via `Mod::Replace` with the new HashSet (or Mod::Delete if the new set is empty).
6. An `AuditEntry` is emitted with action `remove_user_spns`, target_dn = the user DN, and details listing exactly which SPNs were removed and which remain.
7. After successful fix, the `UserDetail` view reloads. If all SPNs were removed, the `Kerberoastable` badge disappears; otherwise the badge remains with the new SPN count in its tooltip.
8. Permission-gated at service layer (`AccountOperator` minimum), same protection as Story 14.4.
9. Defensive: the command refuses to remove an SPN if its prefix is `ldap/`, `host/`, `cifs/`, `HOST/`, `RestrictedKrbHost/` or other system SPNs. The dialog hides these from the selectable list with a "system SPN, cannot be removed safely" tooltip on hover.
10. Unit tests cover the multi-valued attribute manipulation, the system-SPN guard, the empty-after-removal case, and the permission gate.

## Dev Notes

### Architecture

This is the most complex of the three quick-fixes because:
1. The attribute is multi-valued (an LDAP set, not a scalar). `Mod::Replace` with a `HashSet` is the safe way to write multi-valued attributes in `ldap3` - never use `Mod::Delete` per-value, it has gotcha behaviors with the cookie semantics.
2. Some SPNs must never be removed without explicit operator override. Removing `HOST/dc01.corp.local` from a DC computer object would be catastrophic.
3. The fix needs to be partial-success-aware: the operator may select 5 SPNs, 1 turns out to be system-protected, 4 are removed and 1 is reported as blocked.

### System-SPN guard rationale

The list is **conservative** (fewer false positives, more false negatives). If a system SPN is missed and the operator removes it, the consequence is a service outage. If a legitimate SPN is incorrectly classified as system, the operator can't remove it from DSPanel and falls back to ADUC - annoying but safe.

The list is also **easily extensible** - just append to `SYSTEM_SPN_PREFIXES`. Customer feedback during 1.1.0 beta should drive additions. The list is duplicated between Rust and TypeScript (single source of truth would be nice but not worth a build step for a 10-prefix list).

### Multi-valued LDAP attribute semantics

`servicePrincipalName` is multi-valued. To remove specific values, the safe path is:
1. Read the current value list
2. Compute the new list (current minus removed)
3. `Mod::Replace(b"servicePrincipalName", new_set)`

Never use `Mod::Delete(b"servicePrincipalName", removed_set)` because the AD interpretation differs from RFC 4511 in subtle ways (specifically around case-folding of the values in matching). The Replace pattern is what `ldap3`'s test suite uses.

If the resulting set is empty, `Mod::Replace` with an empty HashSet is the right call to delete the attribute entirely; AD accepts that.

### Rust signature

```rust
// src-tauri/src/services/security_indicators.rs (or services/spn.rs)
const SYSTEM_SPN_PREFIXES: &[&str] = &[
    "host",
    "RestrictedKrbHost",
    "cifs",
    "ldap",
    "GC",
    "kadmin",
    "krbtgt",
    "wsman",
    "TERMSRV",
    "MSServerClusterMgmtAPI",
    "MSServerCluster",
    "DNS",
];

pub fn is_system_spn(spn: &str) -> bool {
    spn.split('/')
        .next()
        .map(|prefix| {
            SYSTEM_SPN_PREFIXES
                .iter()
                .any(|sp| prefix.eq_ignore_ascii_case(sp))
        })
        .unwrap_or(false)
}

// src-tauri/src/commands/account.rs
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
#[serde(rename_all = "camelCase")]
pub struct RemoveSpnsResult {
    pub removed: Vec<String>,
    pub kept: Vec<String>,
    pub blocked_system: Vec<String>,
}
```

### Audit entry shape

The audit details string must list exactly what changed so a forensic review can reconstruct state. Format: `removed: [spn1, spn2]; kept: [spn3]; blocked_system: [HOST/dc01.corp.local]`. The lists are stable (alphabetically sorted before formatting) so the audit chain hash is deterministic regardless of input order.
