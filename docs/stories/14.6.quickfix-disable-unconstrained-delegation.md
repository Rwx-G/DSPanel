# Story 14.6: Quick-Fix - Disable Unconstrained Delegation

## Status: Draft

## Story

**As an** Admin,
**I want** a "Disable Unconstrained Delegation" button on a computer object that has the flag set,
**so that** I can remove the most dangerous form of Kerberos delegation (which lets the computer accept TGTs of any user that authenticates against it) in two clicks.

## Acceptance Criteria

1. The `UnconstrainedDelegation` badge on `ComputerDetail` (Story 14.3) gains an inline button "Fix" visible only when the operator's permission level is `Admin` or higher (note: this is a higher bar than user-level fixes since computer delegation changes can break legitimate Kerberos services like SQL Server, IIS with constrained delegation, etc.).
2. Clicking opens a `ConfirmationDialog` warning that disabling unconstrained delegation may break services that rely on Kerberos double-hop. The dialog explains the attack vector (golden ticket TGT replay), recommends migrating to constrained delegation if double-hop is genuinely needed, and includes a checkbox "I have verified no production service on this host requires unconstrained delegation".
3. The dialog goes through `MfaGate` if applicable.
4. A new Tauri command `disable_unconstrained_delegation` is invoked on the computer DN.
5. The backend command reads the current `userAccountControl`, clears bit 0x80000 (TRUSTED_FOR_DELEGATION), writes it back via `Mod::Replace`, and emits an `AuditEntry` with action `disable_unconstrained_delegation`, target_dn = the computer DN, details = previous and new UAC values.
6. After successful fix, the `ComputerDetail` view reloads and the `UnconstrainedDelegation` badge disappears.
7. Permission-gated at service layer (`Admin` minimum).
8. The action is recorded as `Critical` in the audit log severity (it modifies a Kerberos delegation primitive on a domain-joined host, with potential service-impact blast radius).
9. Unit tests cover the UAC bit manipulation and the permission gate.

## Dev Notes

### Architecture

Same shape as Story 14.4 (single-bit UAC flip), but on a computer object with a higher permission gate and a Critical-severity audit entry. The bit is `0x80000` (TRUSTED_FOR_DELEGATION). The contract with `services/security_indicators.rs` (Story 14.1) is symmetric: after the fix, calling `evaluate_computer_indicators` on the same computer no longer returns the `UnconstrainedDelegation` indicator.

### Why Admin and not AccountOperator

Disabling unconstrained delegation on a domain-joined computer can break production services. The blast radius is potentially the entire user population that authenticates against the host. AccountOperator in DSPanel has scope over user account lifecycle; computer delegation primitives are an Admin-level concern. This aligns with the Microsoft AD built-in "Account Operators" group which by design **cannot** modify computer objects.

This decision was flagged as a residual risk during epic planning (epic-14, risk #3): the permission level mapping must align with Story 12.x (RBAC). Verify during implementation that `PermissionLevel::Admin` corresponds to the level the project considers appropriate for computer-object writes. If the project uses a different level for computer writes, align this story to it and document.

### Why Critical severity in the audit log

Three reasons:
1. Reversibility: re-enabling unconstrained delegation requires reading this audit entry to know the previous state. Critical entries are routed to syslog with `severity=4 (warning)` per the existing AuditService convention - that visibility helps SOC teams notice a sysadmin did this.
2. Blast radius: Mistakes here can break entire production stacks until rolled back.
3. Targeting context: an attacker with `Admin` permission would use this exact action to neutralize a defender's Tier 0 delegation hardening. Audit visibility is the defense.

### AuditEntry severity field - small amendment to Epic 11

The current `AuditEntry` struct (per Story 11.1) does not appear to have an explicit `severity` field - severity is inferred from `success: bool` (warning if false, info if true) in the syslog forwarder. This story needs a way to flag entries as Critical. Two options:
- (a) Add an `Option<AuditSeverity>` field (default None, treated as Info). Backwards compatible.
- (b) Use the existing `success: bool` and `details` to imply severity. Hacky.

Recommendation: option (a). The amendment is a small additive change, no breaking impact on existing entries (deserialization with a default works), and unlocks future stories needing fine-grained severity. Document the choice and the Story 11.1 amendment in this story's Change Log.

### Rust signature

```rust
#[tauri::command]
pub async fn disable_unconstrained_delegation(
    state: State<'_, AppState>,
    computer_dn: String,
) -> Result<(), AppError> {
    disable_unconstrained_delegation_inner(&state, &computer_dn).await
}

pub(crate) async fn disable_unconstrained_delegation_inner(
    state: &AppState,
    computer_dn: &str,
) -> Result<(), AppError> {
    if !state
        .permission_service
        .has_permission(PermissionLevel::Admin)
    {
        return Err(AppError::PermissionDenied(
            "Disabling unconstrained delegation requires Admin permission".to_string(),
        ));
    }

    let provider = state.provider();
    let current_uac = provider.get_user_account_control(computer_dn).await
        .map_err(|e| AppError::Directory(e.to_string()))?;
    let new_uac = current_uac & !0x80000u32;
    if new_uac == current_uac {
        return Ok(());
    }
    capture_snapshot(state, computer_dn, "disable_unconstrained_delegation");
    provider.modify_attribute(computer_dn, "userAccountControl", &[new_uac.to_string()]).await
        .map_err(|e| AppError::Directory(e.to_string()))?;
    state.audit_service.record(AuditEntry {
        operator: state.audit_service.operator(),
        action: "disable_unconstrained_delegation".to_string(),
        target_dn: computer_dn.to_string(),
        details: format!("uac: 0x{:X} -> 0x{:X}", current_uac, new_uac),
        severity: Some(AuditSeverity::Critical),
        success: true,
        timestamp: now_iso(),
    });
    Ok(())
}
```
