# Story 2.0: Error Handling Foundation

## Status: Draft

## Story

**As a** developer,
**I want** a unified error handling and network resilience layer,
**so that** all AD write operations (starting in Epic 2) fail gracefully with clear user feedback, automatic retry, and structured audit logging.

## Acceptance Criteria

1. A `DSPanelException` base class exists with typed subclasses: `DirectoryException`, `NetworkException`, `PermissionDeniedException`.
2. All `IDirectoryProvider` LDAP operations wrap raw exceptions into typed `DirectoryException` with user-friendly messages.
3. Transient LDAP/network failures are retried with exponential backoff (1s, 2s, 4s, max 3 attempts) before surfacing the error.
4. A circuit breaker disables a provider after 5 consecutive failures and shows a warning in the notification bar.
5. LDAP connections auto-reconnect after a transient failure (timeout, connection reset) without user intervention.
6. All operations that fail surface a user-friendly error via `INotificationService` (not raw exception messages).
7. A global unhandled exception handler catches unexpected errors, logs them at Error/Fatal level, and shows a generic error notification.
8. Timeout configuration is centralized: LDAP 30s, Graph API 15s, HIBP 5s, WMI 10s.
9. Unit tests cover retry logic, circuit breaker state transitions, exception mapping, and reconnection.

## Dev Notes

### Architecture Pattern

The resilience layer uses the **Decorator pattern**: `ResilientDirectoryProvider` wraps the existing `LdapDirectoryProvider` and adds retry + circuit breaker + exception mapping without modifying the original class. DI registration changes from:

```csharp
services.AddSingleton<IDirectoryProvider, LdapDirectoryProvider>();
```

to:

```csharp
services.AddSingleton<LdapDirectoryProvider>();
services.AddSingleton<IDirectoryProvider>(sp =>
    new ResilientDirectoryProvider(
        sp.GetRequiredService<LdapDirectoryProvider>(),
        sp.GetRequiredService<IRetryPolicy>(),
        sp.GetRequiredService<ICircuitBreaker>(),
        sp.GetRequiredService<INotificationService>(),
        sp.GetRequiredService<ILogger<ResilientDirectoryProvider>>()));
```

### LDAP Error Codes to Handle

| LDAP Error | Code | Classification | Action |
|-----------|------|---------------|--------|
| LDAP_UNAVAILABLE | 52 | Transient | Retry + reconnect |
| LDAP_BUSY | 51 | Transient | Retry |
| LDAP_TIMEOUT | 85 | Transient | Retry + reconnect |
| LDAP_SERVER_DOWN | 81 | Transient | Retry + reconnect |
| LDAP_CONNECT_ERROR | 91 | Transient | Retry + reconnect |
| LDAP_INSUFFICIENT_RIGHTS | 50 | Permanent | Throw PermissionDeniedException |
| LDAP_NO_SUCH_OBJECT | 32 | Permanent | Throw DirectoryException("Object not found") |
| LDAP_CONSTRAINT_VIOLATION | 19 | Permanent | Throw DirectoryException(user-friendly message) |

### Testing Strategy

All resilience components are fully testable without LDAP:
- `ExponentialBackoffRetryPolicy` takes a `Func<Task<T>>` - inject a mock that throws then succeeds
- `CircuitBreaker` is pure state machine logic
- `ResilientDirectoryProvider` takes `IDirectoryProvider` (mock) + `IRetryPolicy` + `ICircuitBreaker`
- Use `Func<int, CancellationToken, Task>` for delay injection (like NotificationService) to avoid real waits in tests
