# Story 2.4: Password Flag Management

## Status: Done

## Story

**As an** Account Operator,
**I want** to toggle password-related flags on user accounts with a dry-run preview,
**so that** I can manage password policies per user while understanding the impact before applying changes.

## Acceptance Criteria

1. The user detail view shows checkboxes for "Password Never Expires" and "User Cannot Change Password", gated to AccountOperator+ permission level.
2. Checkboxes reflect the current state of the user's `userAccountControl` flags.
3. Toggling a checkbox does not immediately apply the change - a "Save" button becomes enabled.
4. Clicking "Save" opens a dry-run preview dialog showing the before/after flag state.
5. The operator must confirm the dry-run preview to apply the change.
6. On success, the UI refreshes to reflect the new flag state.
7. Every flag change - success or failure - is logged via `AuditService` trait with before/after values.
8. Unit tests cover React component logic, dry-run preview, permission gating, and audit logging.

## Dev Notes

### UserAccountControl Flags

The relevant flags in the `userAccountControl` attribute:
- `DONT_EXPIRE_PASSWD = 0x10000` (65536) - password never expires
- `PASSWD_CANT_CHANGE` - this flag is not directly in `userAccountControl`; it is enforced via ACL (deny Change Password to "Everyone" and "SELF"). The implementation must modify the security descriptor rather than the UAC attribute.

### PasswordFlags Model

```typescript
pub struct PasswordFlags(
    bool PasswordNeverExpires,
    bool UserCannotChangePassword);
```

### Dry-Run Preview

The dry-run dialog shows a simple table:

| Flag | Current | Proposed |
|------|---------|----------|
| Password Never Expires | No | Yes |

Only changed flags appear in the preview. If no flags have changed, the "Save" button remains disabled.

### PASSWD_CANT_CHANGE Complexity

Setting "User Cannot Change Password" requires modifying the object's DACL to add deny ACEs for the "Change Password" extended right (GUID `ab721a53-1e2f-11d0-9819-00aa0040529b`) for "Everyone" and "SELF". This is more complex than a simple attribute modify and should be implemented carefully with proper error handling.
