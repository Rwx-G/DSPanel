# Story 3.2: UNC Path Permissions Audit

## Status: Draft

## Story

**As a** HelpDesk operator,
**I want** to enter a UNC path and see its NTFS permissions cross-referenced with two compared users' groups,
**so that** I can quickly determine why one user has access to a share and another does not.

## Acceptance Criteria

1. The comparison view (from Story 3.1) includes a UNC path input field.
2. Entering a UNC path and clicking "Audit" resolves the NTFS ACL for that path.
3. Each ACE is listed with: trustee, access type (Allow/Deny), permissions (Read, Write, FullControl, etc.), and inheritance status.
4. ACEs are cross-referenced with User A's and User B's group memberships, showing a visual access indicator per user (checkmark for access, X for no access, warning for deny).
5. If the UNC path is unreachable or the operator lacks read permissions on the ACL, a clear error message is displayed.
6. Results can be exported to CSV.
7. Unit tests cover ACL resolution, cross-referencing logic, and error handling.

## Dev Notes

### NTFS ACL Resolution

Use `System.IO.DirectoryInfo.GetAccessControl()` or `FileSystemAclExtensions` to retrieve the `DirectorySecurity` object. Iterate `GetAccessRules(true, true, typeof(SecurityIdentifier))` to get both explicit and inherited ACEs.

### AceEntry Model

```csharp
public record AceEntry(
    string TrusteeSid,
    string TrusteeDisplayName,
    AccessControlType AccessType,  // Allow or Deny
    FileSystemRights Permissions,
    bool IsInherited);
```

### Cross-Reference Logic

For each ACE, check if the trustee SID matches any of the user's group SIDs (from `tokenGroups`) or the user's own SID. This determines whether the ACE applies to that user.

Deny rules take precedence over Allow rules when computing effective access.

### UNC Path Validation

Validate that the input starts with `\\` and contains at least a server and share name. Reject local paths and URLs.
