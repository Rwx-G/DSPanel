# Story 8.4: Remote Workstation Monitoring

## Status: Draft

## Story

**As a** HelpDesk operator or administrator,
**I want** to view real-time system metrics for a remote workstation from the computer lookup view,
**so that** I can diagnose performance issues without physically accessing the machine or using separate tools.

## Acceptance Criteria

1. When viewing a computer's details, a "Monitoring" panel shows CPU usage, RAM usage, active sessions, running services, and disk space.
2. Metrics are collected via WMI/CIM remote queries to the target workstation.
3. The monitoring panel auto-refreshes every 5 seconds (configurable).
4. If the workstation is unreachable or WMI access is denied, the panel degrades gracefully with a clear status message.
5. Individual metric sections show "unavailable" if a specific query fails while others succeed.
6. Unit tests cover WMI query construction, metric parsing, error handling, and auto-refresh.

## Dev Notes

### WMI Query Implementation

```typescript
// Using windows crate WMI bindings (Windows-only)
var scope = new WMI scope via windows crate($"\\\\{hostname}\\root\\cimv2");
scope.Options.Timeout = TimeSpan.FromSeconds(5);

var cpuQuery = new ObjectQuery("SELECT LoadPercentage FROM Win32_Processor");
using var cpuSearcher = new WMI query via windows crate(scope, cpuQuery);
var cpuResults = cpuSearcher.Get();

// Using windows crate CIM bindings (CIM, preferred)
var session = CIM session via windows crate.Create(hostname);
var cpuInstances = session.QueryInstances("root/cimv2", "WQL",
    "SELECT LoadPercentage FROM Win32_Processor");
```

### SystemMetrics Model

```typescript
pub struct SystemMetrics
{
    public double CpuUsagePercent { get; init; }
    public long TotalMemoryMb { get; init; }
    public long UsedMemoryMb { get; init; }
    public IReadOnlyList<DiskInfo> Disks { get; init; }
    public IReadOnlyList<ServiceInfo> Services { get; init; }
    public IReadOnlyList<SessionInfo> Sessions { get; init; }
    public DateTime Timestamp { get; init; }
}
```

### Performance Considerations

- WMI queries can be slow over the network - use async and timeouts
- Service list can be large - consider filtering to auto-start services by default
- Cancel inflight queries when the user navigates away to avoid resource leaks
- Use `cancellation token` throughout the query pipeline
